### 4.5 Assembler output byte contract

The shipped `bl` has a deterministic byte shape, fixed by `scripts/assemble-bl.sh` at commit `d675e5b`. M5.5 MUST NOT modify the assembler. The assembled bytes are:

```
Line 1:  #!/bin/bash\n
Line 2:  # GENERATED FILE ‚Äî edit src/bl.d/NN-*.sh and run "make bl"\n
Line 3:  # Source of truth: src/bl.d/ ‚Äî see CLAUDE.md "bl source layout" rule.\n
Line 4:  # Do not edit bl directly; make bl-check will fail in CI.\n
<blank>
Line 6:  # --- src/bl.d/00-header.sh ---\n
...      <00-header.sh contents, with any leading shebang stripped>
<blank>
Line N:  # --- src/bl.d/10-log.sh ---\n
...      <10-log.sh contents, with any leading shebang stripped>
...      [repeat for each of the 14 parts, numeric-sorted]
```

**Separator detail:** The separator between part bodies is `\n# --- <path> ---\n` (blank line before the `# ---` marker) ‚Äî this is what `printf '\n# --- %s ---\n' "$p"` emits. Each part body is `sed '1{/^#!/d;}'`-piped, so any optional part-local shebang is dropped.

**Trailing newline:** Each part file ends with a trailing newline (bash convention). The assembler does not add an extra trailing newline after the last part; the assembled `bl` ends with whatever the last part (`90-main.sh`) ends with. Engineer MUST ensure `90-main.sh` ends with `main "$@"` followed by exactly one newline.

**Executable bit:** `make bl` runs `chmod +x bl` after assembly. `bl` is committed with executable bit set (git preserves mode).

### 4.6 Drift-guard semantics

`make bl-check` (Makefile at `d675e5b`) has three states:

| State | Condition | Behavior | Exit |
|---|---|---|---|
| A ‚Äî pre-M5.5 compat | `src/bl.d/[0-9]*.sh` glob empty (`BL_PARTS` unset) | Print "`bl-check: src/bl.d/ not populated yet (pre-M5.5) ‚Äî skipping drift check`"; pass | 0 |
| B ‚Äî post-M5.5 clean | Parts populated AND `assemble-bl.sh` output matches `bl` byte-exact | Print "`bl-check: in sync`"; pass | 0 |
| C ‚Äî drift | Parts populated AND assembly differs from committed `bl` | Print "`bl-check: src/bl.d/ parts drifted from bl ‚Äî run 'make bl' and re-commit`" to stderr; fail | 1 |

M5.5 transitions the repo from state A to state B. State C is the permanent failure mode that catches future "edited `bl` directly" bugs.

**CI reach (post-M5.5):** Once the `tests/Makefile` wiring (¬ß7.4) lands, `make -C tests test` invokes `_bl-drift-check` ‚Üí `make -C .. bl-check` before Docker build. State C fails the test run before any BATS runs ‚Äî fast feedback on drift. Pre-M5.5, `bl-check` was reachable only by running `make bl-check` manually at repo root.

### 4.7 Shellcheck parity protocol

**Baseline (pre-M5.5):** `shellcheck bl` on the pre-refactor `bl` (HEAD before M5.5 commit) exits 0 with zero output. Captured: `shellcheck bl; echo $?` ‚Üí `0`, stdout+stderr empty.

**Post-M5.5 criterion:** `shellcheck bl` on the regenerated `bl` must also exit 0 with zero output. Line numbers will shift; finding set is empty both before and after, so parity is trivial.

**If post-M5.5 shellcheck finds anything:** MUST-FIX blocker. The likely causes are:
- A carve artifact reintroduced an earlier bug (e.g., unquoted var restored by sed slip).
- A separator-line interaction with a multi-line bash construct (unlikely; `# --- ---` is a comment).
- An unintended whitespace change that changed a variable reference's context (shellcheck is line-sensitive).

Recovery: bisect by regenerating `bl` from fewer parts (temporarily rename some parts to non-matching names so assembler skips them) until the offending part is identified. Fix the part; re-run.

**Shellcheck directive hoisting:** The pre-refactor `bl` has `# shellcheck disable=SC2094` on line 2 as a file-global directive. `SC2094` handles in-loop file-handle manipulation (`M4` observe handlers read log files while concurrently cleaning up temp paths ‚Äî false positive). This directive MUST appear in `00-header.sh` (not in a later part) so it remains a file-global directive in assembled `bl`.

Similarly, per-constant `# shellcheck disable=SC2034` directives on `BL_EX_SCHEMA_VALIDATION_FAIL`, `BL_EX_TIER_GATE_DENIED`, `BL_EX_CONFLICT`, `BL_EX_NOT_FOUND` (constants defined for M4+/M5+/M7/M8 consumers but not yet consumed in M5) MUST be preserved inline with their declarations ‚Äî they are above each `readonly` line in the pre-refactor `bl` and must travel together into `00-header.sh`.

## 6. Conventions

### 6.1 Part-file shebang policy

Part files MAY have a `#!/bin/bash` on line 1 for editor syntax highlighting. The assembler strips it (`sed -e '1{/^#!/d;}'`). Part files MUST NOT have other leading-line content that could confuse the assembler (no encoding declarations, no env-hashbang variants).

### 6.2 `set -euo pipefail` is declared exactly once

Only `00-header.sh` contains `set -euo pipefail`. Repeating it in later parts is harmless but inflates diffs. Engineer MUST NOT redeclare in later parts.

### 6.3 Section-header comments

Pre-refactor `bl` has `# ----...---- / # <Section title>` section-header comments between function blocks. These are preserved verbatim within their new part file. If a section-header ends up at the top of a part file (e.g., `# Logging helpers` at the top of `10-log.sh`), that's fine ‚Äî readable.

### 6.4 Inline shellcheck disables

`# shellcheck disable=SCxxxx  # <rationale>` comments stay with their target line. Example: the four `# shellcheck disable=SC2034` directives above `BL_EX_SCHEMA_VALIDATION_FAIL`, `BL_EX_TIER_GATE_DENIED`, `BL_EX_CONFLICT`, `BL_EX_NOT_FOUND` (declared but not yet consumed in M5) travel with the `readonly` lines into `00-header.sh`.

### 6.5 Path references inside part files

Part files MUST NOT reference their own path or sibling parts. A helper in `10-log.sh` cannot `source ../20-api.sh` or `readlink -f "$0"` to find itself. All cross-part calls go through bash function names, resolved at `main "$@"` time by late-binding.

### 6.6 Coreutils `command` prefix

Pre-refactor `bl` already uses `command <util>` prefix for portability (per proj/CLAUDE.md ¬ßShell Standards "Coreutils `command` prefix"). Part files inherit this ‚Äî no change needed in carve.
