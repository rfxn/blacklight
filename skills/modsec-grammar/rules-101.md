# modsec-grammar — SecRule reference for synthesized defenses

Loaded by the router on every `bl consult --synthesize-defense` call that targets ModSecurity. This is the grammar floor — variables, operators, actions, phases, and chaining — that the synthesis call must respect to produce rules that actually parse and load. Pair with `defense-synthesis/modsec-patterns.md` for the operator-validated rule shapes; this file is the language reference, that one is the idiom catalog.

Note: ModSec `tag:'application-attack'` strings in this file are rule-engine grammar literals (OWASP CRS convention) — preserved verbatim for tool compatibility.

## SecRule directive shape

Every ModSec v2 and v3 rule follows the same five-token structure:

```
SecRule VARIABLES OPERATOR ACTIONS
```

- `VARIABLES` — what to inspect, pipe-separated when multiple. Example: `REQUEST_URI|ARGS|REQUEST_HEADERS:User-Agent`.
- `OPERATOR` — how to test the variable values. Quoted string starting with `@` for a named operator, or a bare regex for the implicit `@rx`. Example: `"@rx (?:union\s+select|select.+from)"`.
- `ACTIONS` — comma-separated list of what to do on match. Quoted. Example: `"id:1001,phase:2,deny,log,msg:'SQLi attempt',status:403"`.

The directive must be on logical line. Continuation uses `\` at end of line.

ModSec v3 (`libmodsecurity` + connector) and ModSec v2 (`mod_security2`) share the same `SecRule` syntax. Differences are in available operators (v3 adds `@inspectFile`, drops some legacy operators), variable scope, and the configuration directives outside `SecRule` itself. Rules written to v2 grammar generally load on v3; the inverse is not always true. The synthesis call must target v2.9+ as the floor unless the case's manifest is explicitly v3-tagged.

## Phases — and which to use for what

Five processing phases. Each phase has a fixed scope of variables that are defined at that point in the request lifecycle.

- **phase:1** — request headers parsed, body not yet read. Variables: `REQUEST_HEADERS`, `REQUEST_METHOD`, `REQUEST_URI`, `REQUEST_FILENAME`, `REMOTE_ADDR`, `ARGS_GET`. Use for: cheap blocks based on URL, method, source IP, header content. Do not reference `REQUEST_BODY` or `ARGS_POST` in phase:1 — they are not yet populated.
- **phase:2** — request body parsed (form-urlencoded, multipart, JSON, XML based on `SecRequestBodyAccess` and parser config). Variables: everything from phase:1 plus `REQUEST_BODY`, `ARGS_POST`, `ARGS` (combined GET+POST), `FILES`, `FILES_NAMES`, `XML`, `JSON`. Use for: payload inspection, form-field rules, file-upload rules. Default phase for most application-layer rules.
- **phase:3** — response headers parsed, body not yet generated by upstream. Variables: `RESPONSE_HEADERS`, `RESPONSE_STATUS`. Use for: blocks based on response code or headers (e.g., rewrite an upstream `5xx` to a generic page).
- **phase:4** — response body buffered (subject to `SecResponseBodyAccess`). Variables: `RESPONSE_BODY`, plus everything earlier. Use for: data-loss prevention, blocking responses that contain sensitive patterns. Expensive — buffers the response.
- **phase:5** — logging. Variables: full request and response. No actions can change the response. Use for: conditional audit-log inclusion, accounting.

Rule of thumb: place a rule in the earliest phase where the variables it needs are defined. A URL-block rule in phase:2 wastes the body parse; a body-content rule in phase:1 will not match because the body is not yet read.

## Variable catalog (the load-bearing subset)

- `REQUEST_URI` — full request URI including query string.
- `REQUEST_URI_RAW` — pre-normalization variant; useful when normalization itself is the attack surface.
- `REQUEST_FILENAME` — path component only, no query string.
- `REQUEST_BASENAME` — final path component (`shell.php`).
- `REQUEST_METHOD` — `GET`, `POST`, etc.
- `REQUEST_PROTOCOL` — `HTTP/1.1`.
- `REQUEST_HEADERS` — collection. Index by name: `REQUEST_HEADERS:User-Agent`. Names are case-insensitive.
- `REQUEST_HEADERS_NAMES` — collection of header names only.
- `REQUEST_BODY` — full body when parsed as a single string (urlencoded body or text/plain).
- `REQUEST_COOKIES`, `REQUEST_COOKIES_NAMES` — cookie collections.
- `ARGS` — combined GET+POST arg collection. `ARGS:foo` is the `foo` parameter; `ARGS_NAMES` is the set of parameter names.
- `ARGS_GET`, `ARGS_POST` — split variants.
- `FILES`, `FILES_NAMES`, `FILES_SIZES`, `FILES_TMPNAMES` — multipart upload metadata.
- `REMOTE_ADDR` — client IP as the web server sees it. Behind a proxy, this is the proxy unless `mod_remoteip` (Apache) or equivalent is configured.
- `REMOTE_HOST` — reverse DNS, expensive, usually unused.
- `RESPONSE_STATUS`, `RESPONSE_HEADERS`, `RESPONSE_BODY` — phase:3+ response inspection.
- `TX:foo` — per-transaction variable, scoped to one request. Use for state across chained rules.
- `IP:foo` — per-source-IP persistent variable (requires `SecAction "initcol:IP=%{REMOTE_ADDR}"`). Use for rate-limiting state.

## Operator catalog

- `@rx <regex>` — PCRE regex match. The default if the operator is omitted. Case-sensitive unless `t:lowercase` transformation is applied.
- `@beginsWith <string>` — literal prefix match. Cheaper than `@rx` for fixed prefixes.
- `@endsWith <string>` — literal suffix match.
- `@contains <string>` — literal substring match.
- `@containsWord <string>` — substring match with word boundaries.
- `@streq <string>` — full string equality.
- `@within <string>` — checks whether the variable value is within the operator string (inverted `@contains`).
- `@eq <int>`, `@gt <int>`, `@ge <int>`, `@lt <int>`, `@le <int>` — numeric comparisons.
- `@ipMatch <ip-or-cidr,...>` — match against IP or CIDR list. Example: `@ipMatch 10.0.0.0/8,192.168.0.0/16`.
- `@ipMatchFromFile <path>` — same as `@ipMatch` but loaded from a file, one entry per line.
- `@pmFromFile <path>` — phrase match from file (Aho-Corasick), one phrase per line. Fast for large literal lists.
- `@pm <phrase> <phrase>...` — inline phrase match.
- `@validateByteRange <range>` — byte values in variable must fall in the range; useful for character-set restrictions.
- `@detectSQLi`, `@detectXSS` — libinjection-backed detectors (v3, and v2 with libinjection compiled in).
- `@inspectFile <script>` — invoke an external script against an uploaded file (v3).

Negation: prefix with `!`. Example: `"!@ipMatch 10.0.0.0/8"` matches when `REMOTE_ADDR` is not in the range.

## Actions catalog

Disruptive actions (one per rule):

- `deny` — block the request. Pair with `status:<code>` for the response code (default 403).
- `drop` — close the connection without responding.
- `redirect:<url>` — issue a redirect.
- `proxy:<url>` — proxy the request to a different backend (rarely used).
- `pass` — explicit no-block (default when no disruptive action is named).

Non-disruptive actions (compose freely):

- `id:<int>` — required. Unique numeric rule id. Range conventions: 1-99999 reserved for OWASP CRS, 100000+ for site-local. Pick a range scheme and stick to it.
- `phase:<1-5>` — required. Phase to evaluate the rule in.
- `log` / `nolog` — write to error log on match (or not).
- `auditlog` / `noauditlog` — include in the audit log on match (or not).
- `msg:'<string>'` — message text written to the log on match.
- `tag:'<string>'` — tag the rule for grouping in reports. Repeat for multiple tags.
- `severity:<0-7>` — log severity (0=emergency, 7=debug).
- `status:<code>` — response status code paired with `deny`.
- `chain` — link to the next `SecRule` directive; both must match for the disruptive action to fire.
- `skip:<n>` — skip the next `n` rules on match.
- `skipAfter:<marker>` — skip ahead to a `SecMarker` directive.
- `setvar:<var>=<value>` — set a transaction or persistent variable. `setvar:tx.score=+5` to increment.
- `expirevar:<var>=<seconds>` — schedule a persistent variable for expiry.
- `ctl:<directive>=<value>` — runtime configuration change. Common: `ctl:ruleRemoveById=<id>`, `ctl:auditEngine=Off`, `ctl:requestBodyAccess=On`.

Transformations (apply to the variable before the operator runs, left-to-right):

- `t:none` — remove all default transformations. Use first to start clean.
- `t:lowercase` — fold to lowercase.
- `t:uppercase` — fold to uppercase.
- `t:trim` — strip leading/trailing whitespace.
- `t:urlDecode` — decode `%xx` sequences.
- `t:urlDecodeUni` — decode `%xx` and `%uXXXX`.
- `t:htmlEntityDecode` — decode HTML entities.
- `t:base64Decode` — decode base64.
- `t:compressWhitespace` — collapse runs of whitespace to single space.
- `t:removeNulls` — strip `\0` bytes.
- `t:normalizePath` — collapse `..` and `//` in paths.

Chain transformations to defeat known evasion patterns: `t:none,t:urlDecodeUni,t:lowercase,t:compressWhitespace`.

## Chained rules

A `chain` action links the current rule to the immediately following `SecRule`. The chain matches only when every rule in the chain matches; the disruptive action and most non-`chain` actions belong on the head rule, not the chained continuations.

```
SecRule REQUEST_URI "@beginsWith /admin/auth/login" \
    "id:200001,phase:2,deny,log,msg:'admin login flood',chain"
    SecRule IP:auth_attempts "@gt 20"
```

Chained rules share the transaction state; counters set with `setvar:tx.foo=...` in one rule are visible in subsequent rules in the same transaction. Use chains to express compound conditions ("URL is admin login AND request body contains pattern AND source IP not in trust list").

## Exception patterns

Disabling a rule for a known-benign caller is a first-class operation; rules that fire on legitimate traffic must not be left in `log,pass` mode permanently because that hides real matches.

- **SecRuleRemoveById** — at config-load time, remove a rule globally. Use sparingly; loses the rule for everyone.
- **SecRuleRemoveByTag** — remove rules by tag.
- **SecRuleUpdateActionById** — change the actions on a rule without removing it. Example: `SecRuleUpdateActionById 200001 "pass,nolog"` to silence a rule.
- **`<LocationMatch>` block** — Apache config wrapper. Apply `SecRuleRemoveById` only inside a path. Example: `<LocationMatch "^/api/upload"> SecRuleRemoveById 949110 </LocationMatch>`.
- **Conditional skip** — earlier rule sets a `tx.` variable; a `chain` rule keys on it and applies `ctl:ruleRemoveById=...` for the rest of the request.
- **`SecMarker` + `skipAfter`** — define a named marker, skip ahead to it on a condition. Useful for whole sections of rules that should not fire for, e.g., a vendor traffic class.

## Audit log

`SecAuditEngine` controls whether the audit log records anything. `On` records every transaction; `RelevantOnly` records only transactions that triggered a rule (the typical production setting); `Off` disables.

`SecAuditLogParts` selects which sections of each transaction record are written. Letters are part identifiers:

- `A` — audit log header.
- `B` — request headers.
- `C` — request body.
- `E` — original response body (modified or matched against).
- `F` — response headers.
- `H` — audit log trailer (rule matches, processing time).
- `I` — reduced multipart request body.
- `J` — multipart files info.
- `K` — list of every rule that matched, in matching order.
- `Z` — final boundary.

Production typical: `SecAuditLogParts ABIJDEFHKZ` or a tighter subset; record everything during incident response, trim afterward.

`SecAuditLogType` of `Concurrent` writes one file per transaction under `SecAuditLogStorageDir`, indexed by `SecAuditLog`. `Serial` writes a single file. Concurrent scales better.

## Worked examples

Each example is annotated; rule ids are illustrative.

**1 — URL-evasion: image extension routing to PHP**

The APSB25-94-class pattern routes a `.jpg`/`.png`/`.gif` request to PHP via a rewrite or `auto_prepend_file`. Block requests whose path looks like an image but whose response would be PHP-handled.

```
SecRule REQUEST_FILENAME "@rx \.(?:jpg|jpeg|png|gif|webp)$" \
    "id:100101,phase:1,chain,deny,status:403,log,\
     msg:'image-extension request reaching PHP handler',\
     tag:'application-attack',tag:'apsb25-94-class'"
    SecRule REQUEST_HEADERS:Accept "@rx (?:^|,)\s*application/x-php" \
        "t:none,t:lowercase"
```

The chain here is illustrative; in practice the second-rule key depends on how the host distinguishes image-handler-bound from PHP-handler-bound requests (location block, file ownership, server-status response). Operator validation required before deploy.

**2 — Credential harvest: rate-limit admin login**

Block more than 20 admin-login POSTs from one IP per minute. State held in IP collection.

```
SecAction "id:100200,phase:1,nolog,pass,initcol:IP=%{REMOTE_ADDR}"

SecRule REQUEST_URI "@rx ^/[^/]+/admin/auth/login" \
    "id:100201,phase:2,chain,pass,log,\
     msg:'admin login attempt counted'"
    SecRule REQUEST_METHOD "@streq POST" \
        "setvar:IP.login_attempts=+1,expirevar:IP.login_attempts=60"

SecRule IP:login_attempts "@gt 20" \
    "id:100202,phase:1,deny,status:429,log,\
     msg:'admin login flood from %{REMOTE_ADDR}',\
     tag:'application-attack',tag:'rate-limit'"
```

Chain on rule 100201 ensures the counter only increments for `POST` requests against the admin-login URL. Rule 100202 fires in phase:1 of the next request from the same IP, before the body is parsed.

**3 — Generic POST size limit on a known-safe endpoint**

Cap the request body size for a specific path. Avoids the global `SecRequestBodyLimit` becoming the only knob.

```
SecRule REQUEST_URI "@beginsWith /api/v1/contact" \
    "id:100301,phase:1,chain,deny,status:413,log,\
     msg:'oversized POST to contact endpoint'"
    SecRule REQUEST_HEADERS:Content-Length "@gt 8192"
```

Notice the rule lives in phase:1 — `Content-Length` header is available before the body is read, so the block fires before any body bytes are parsed.

**4 — Chained rule with location-scoped exception**

Block uploads of files whose name ends in `.php`, except inside the admin extension-upload endpoint where vendor packages legitimately contain `.php`.

```
SecRule FILES_NAMES "@rx \.php$" \
    "id:100401,phase:2,deny,status:403,log,\
     msg:'PHP file upload blocked: %{MATCHED_VAR}',\
     tag:'application-attack'"

<LocationMatch "^/[^/]+/admin/admin/setupwizard/extensionmanager/install">
    SecRuleRemoveById 100401
</LocationMatch>
```

The `<LocationMatch>` exception is operator territory — the path pattern depends on the chosen admin frontName; an unscoped exception is the failure mode that turns the rule into a no-op.

## Rule id discipline

Pick a numeric range and document it. A common convention: `100000-199999` for site-local rules, `200000-299999` for tenant-specific rules in a shared-hosting context, leaving the OWASP CRS range (`900000-999999`, `9xxx`) untouched. A duplicate `id:` across loaded rule files is a load-time error in v3 and a silent override in v2 — always grep the full rules tree before allocating a new id.

## OWASP Core Rule Set

Most production deployments load the OWASP CRS (`https://github.com/coreruleset/coreruleset`) as a baseline. Site-local rules should layer on top, not replicate CRS coverage. When generating a new rule, check whether the evidence-pattern class is already covered by a CRS rule (the `crs-setup.conf` and `rules/REQUEST-*.conf` files are the index) and prefer tuning CRS via `ctl:ruleRemoveById` and `crs-setup.conf` variables over writing a parallel rule.
