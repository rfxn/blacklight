## host-5

Staged compromise: Magecart-class JS skimmer at `pub/media/catalog/product/.cache/skimmer.php`. **The drop path matches the PolyShell convention** so the fs_hunter fires the same `unusual_php_path` category as host-2/4/7 — initial signal looks like the same campaign. **The intent reconstructor reveals the family-marker divergence.** Distinct C2 (`skimmer-c2.example`); no shared infrastructure with the PolyShell campaign. **Sim-day 14.**

### fs_hunter should find

- **Category:** `unusual_php_path`
  **Source ref:** `fs/var/www/html/pub/media/catalog/product/.cache/skimmer.php`
  **confidence:** 0.7 or higher
  *Note:* this is the same category and same path-shape as host-2; the family divergence is invisible to fs_hunter and only surfaces during intent reconstruction.

### log_hunter should find

- **Category:** `suspicious_outbound`
  **Finding shape:** repeated GETs from `203.0.113.71` to `skimmer.php` with `(skimmer-c2.example)` in the user-agent
  **Source refs:** log line numbers
  **confidence:** 0.7 or higher
- *No `url_evasion` findings* — skimmer family fetches its dropper directly, not via double-extension paths. This is itself a divergence signal but the category name is just absence rather than a positive finding.

### timeline_hunter should find

- **Category:** `causal_adjacency` — fs finding on `skimmer.php` + log finding on the same path within seconds.

### Intent reconstructor (Opus 4.7) returns

- `family`: NOT a PolyShell variant. Family-marker divergence:
  - No `chr()` ladder, no 4-chunk base64 concatenation
  - No `gzinflate` / `base64_decode` chain
  - No `$_POST['c']` command dispatch table
  - No callback to `vagqea4wrlkdg.top` — different C2 (`skimmer-c2.example`)
  - Payload is client-side JavaScript (Magecart-class), not server-side PHP eval
- `inferred` capabilities: `payment-data-exfil-capability-staged`, `js-injection-into-checkout-template`
- `likely_next`: `skimmer-injection-into-checkout` (rank top); `lateral-target-selection` against the same Magento admin credential set (rank lower)

### Hypothesis change (sim-day 14, on existing `CASE-2026-0007`)

- `case_engine.revise()` called with the host-5 evidence + the intent reconstructor's family-marker output.
- `support_type`: **`unrelated`**
- `revision_warranted`: `False`
- `open_questions_additions`: includes "host-5 evidence may belong to a separate investigation; recommend split"

### Orchestrator action: case split

- Orchestrator detects `support_type == "unrelated"`, allocates new case_id via `_allocate_case_id()` → returns `CASE-2026-0008` (or next sequential per allocator rules — first call after the Day-2 baseline 0007 returns 0008).
- Calls `case_engine.split_case(prior=CASE-2026-0007, new_evidence=[host-5 rows], new_case_id=CASE-2026-0008)`.
- Writes both case files atomically:
  - `CASE-2026-0007`: `split_into = [CASE-2026-0008]`; `last_updated_at` and `updated_by` refreshed by `case_engine.split_case`. `hypothesis.history` is **NOT** mutated by the split path — split is a structural operation, not a hypothesis revision. The split-out reason lives in the operator-readable `split_into` cross-reference; revision history stays the domain of `case_engine.revise()` + `apply_revision()`. Status remains `active` (the PolyShell campaign continues).
  - `CASE-2026-0008`: `merged_from = [CASE-2026-0007]`; `hypothesis.current.confidence`: initial-template default; `hypothesis.history`: empty (initial state — first revise() call against this new case will populate).

### Negative assertions

- No mutation of `CASE-2026-0007`'s `hypothesis.history` entries by the split path (split path does not touch history; `case_engine.split_case` only mutates `split_into`, `last_updated_at`, `updated_by`).

## Case-split scenario

This section documents the demo-time mind-change moment and how to verify it on disk after a sim run.

### Pre-state

- `CASE-2026-0007`: active, hypothesis confidence ~0.85, `evidence_threads` includes hosts 2, 4, 7. PolyShell campaign attribution with shared C2 `vagqea4wrlkdg.top`.

### Trigger (sim-day 14)

The host-5 report arrives. Three signals interact:

1. `fs_hunter` flags `unusual_php_path` on `pub/media/catalog/product/.cache/skimmer.php` — same path SHAPE as host-2's `a.php`, so initial filesystem evidence looks like the same campaign.
2. `log_hunter` flags `suspicious_outbound` with the C2 string `skimmer-c2.example` — already a divergence from `vagqea4wrlkdg.top`, but log evidence alone is not the gate.
3. `intent_reconstructor` (Opus 4.7 + adaptive thinking) reads `skimmer.php` and reports family-marker divergence: no `chr()` ladder, no 4-chunk base64, no gzinflate, no `$_POST['c']` dispatch table; payload is client-side JavaScript skimmer, not server-side PHP eval. Family ≠ PolyShell.

### Action

- `case_engine.revise()` called against `CASE-2026-0007` with the host-5 evidence + intent output.
- Returns `support_type=unrelated`, `revision_warranted=False`.
- Orchestrator detects `unrelated` and routes through the split branch:
  - `_allocate_case_id()` → `CASE-2026-0008`
  - `case_engine.split_case(prior=CASE-2026-0007, new_evidence=[host-5 rows], new_case_id=CASE-2026-0008)`
  - Writes both case files atomically.

### Post-state

```bash
ls curator/storage/cases/
# CASE-2026-0007.yaml  CASE-2026-0008.yaml

cat curator/storage/cases/CASE-2026-0007.yaml | yq '.split_into'
# - CASE-2026-0008

cat curator/storage/cases/CASE-2026-0007.yaml | yq '.hypothesis.history | length'
# (whatever count existed before the split — unchanged; split path does not append history)

cat curator/storage/cases/CASE-2026-0008.yaml | yq '.merged_from'
# - CASE-2026-0007

cat curator/storage/cases/CASE-2026-0008.yaml | yq '.hypothesis.history | length'
# 0  (initial state; first revise() call against the new case populates)
```

### Demo narration mapping (Variant B Beat 3.5, `demo/script.md`)

The on-screen visual lingers on:
1. The split YAML files appearing side-by-side
2. The `CASE-2026-0007.split_into` and `CASE-2026-0008.merged_from` cross-references
3. The case engine's reasoning text citing the family-marker divergence

This is the "judgment didn't merge them" moment — pattern matching on the path would have appended host-5 to the PolyShell case; the family-marker check refused.

## How this file is used

- **Day 2 live smoke (Phase 9):** operator reads the host-2 section before running the orchestrator; after running, compares `sqlite3 curator/storage/evidence.db 'SELECT hunter, category, finding FROM evidence'` against the expected shapes.
- **Day 3 revision test:** the hypothesis-revision unit test fixtures reference categories from this file. Drift here means fixture drift there.
- **Day 5 demo recording:** the time-compression simulator (`demo/time_compression.py`) plays evidence reports for hosts 2, 4, 5, 7 across sim-days 1, 5, 14, 7 (Variant B order); host-1's anticipatory block fixture is replayed at sim-day 14. The case-split section above defines the on-disk verification commands the operator runs immediately after the recording.
- **Demo script:** the 0:35 beat ("Case opens: CASE-2026-0007, confidence 0.4...") echoes the host-2 initial-hypothesis shape verbatim. The 1:55 beat (Variant B mind-change) echoes the case-split scenario above.
